# Orderlynk — Backend

Spring Boot 4 (Java 21) REST API for Orderlynk: the vendor operating system and
commerce rails (orders, payments, fulfillment, payouts) built to the product spec.
The React SPA lives in a separate repo and talks to this API.

## Quick start (`http://localhost:8080`)

Needs **JDK 21** and a **MySQL 8** (or TiDB) instance. Start MySQL locally with Docker:

```bash
docker run --name orderlynk-mysql -e MYSQL_ROOT_PASSWORD=orderlynk \
  -e MYSQL_DATABASE=orderlynk -p 3306:3306 -d mysql:8
```

Then run the app from this directory:

```bash
./mvnw spring-boot:run
```

Connection defaults to `jdbc:mysql://localhost:3306/orderlynk`, user `root`,
password `orderlynk` — override with `DB_URL` / `DB_USERNAME` / `DB_PASSWORD`.
Liquibase creates the schema on startup; demo data is seeded outside the `prod`
profile.

## Architecture (`com.myorderlynk.app`)

| Layer        | Package      | Responsibility                                                         |
| ------------ | ------------ | ---------------------------------------------------------------------- |
| Domain       | `domain`     | JPA entities + `enums` for every option set                            |
| Persistence  | `repo`       | Spring Data JPA repositories                                           |
| Services     | `service`    | Business logic: fees, fulfillment flows, payouts, notifications, audit |
| Web          | `web`        | REST controllers + global error handling                              |
| DTOs         | `dto`        | Request/response records + entity → DTO `Mapper`                       |
| Security     | `security`   | JWT issuing/validation (OAuth2 resource server), role-based access     |
| Config       | `config`     | `DataSeeder` (demo data, non-prod profile)                             |

Key design points:

- **Auth** — email/password login issues an HS256 JWT (`roles`, `vendorId` claims),
  validated by Spring Security's OAuth2 resource server. Access is role-based
  (`CUSTOMER` / `VENDOR` / `ADMIN`) via `@PreAuthorize`, plus per-resource ownership
  checks (a vendor can only touch their own products/orders).
- **Fee engine** (`FeeCalculator`):
  `Total = Subtotal + Logistics + Platform + Processing`, with vendor payable,
  logistics payable and platform revenue tracked as an internal ledger per order.
- **Fulfillment flows** (`FulfillmentFlows`) encode the distinct status sequences per
  fulfillment type and validate transitions. Reaching `READY_FOR_PICKUP` generates a
  pickup code.
- **Order IDs** use the `OB-YYMMDD-RANDOM` format.
- **Audit + notifications** — every payment/fulfillment change is written to
  `StatusChangeLog`; notifications are recorded to `NotificationLog` (the seam for real
  email/WhatsApp later).
- **Payouts** — `PayoutService` builds weekly vendor reconciliation reports.

**Primary keys** are `UUID` (generated by Hibernate, stored as `CHAR(36)`), exposed as
strings over the API.

## Database & migrations

**Liquibase** owns the schema (`ddl-auto: none`). The changelog is at
`src/main/resources/db/changelog/` — a master file including
`changes/001-initial-schema.xml` and `changes/002-seed-admin.xml`, run automatically
on startup. The baseline is **dialect-portable** (MySQL, TiDB, and H2 for tests) and
TiDB-friendly: string UUID keys (no `AUTO_INCREMENT`), enums as `VARCHAR`, and no
DB-level foreign keys (relationships are id-column + index). A precondition lets the
baseline adopt a database that already has the tables. Add future changes as new
`changes/00N-*.xml` files included from the master changelog.

**TiDB:** TiDB speaks the MySQL protocol, so point the same MySQL JDBC driver at your
TiDB endpoint (TiDB Cloud listens on port `4000`), e.g.
`DB_URL=jdbc:mysql://gateway.tidbcloud.com:4000/orderlynk?useSSL=true&serverTimezone=UTC`.

**Profiles:** default = MySQL/TiDB with demo seeding. `prod` = no seeding — set
`SPRING_PROFILES_ACTIVE=prod` and override `DB_URL` / `DB_USERNAME` / `DB_PASSWORD`,
`APP_JWT_SECRET`, `APP_CORS_ORIGINS`, `APP_PUBLIC_BASE_URL`. The `test` profile runs
Liquibase on in-memory H2 (MySQL mode), so `./mvnw test` needs no database.

## API surface (selected)

| Method & path                              | Access   | Purpose                                  |
| ------------------------------------------ | -------- | ---------------------------------------- |
| `POST /api/auth/register` · `login`        | public   | Account creation / login → JWT           |
| `POST /api/auth/change-password`           | auth     | Rotate your own password (any role)      |
| `GET  /api/storefronts` · `/{slug}`        | public   | Marketplace list / storefront            |
| `POST /api/orders/quote`                   | public   | Fee quote before checkout                |
| `POST /api/orders`                         | public   | Guest/customer checkout                  |
| `POST /api/orders/track`                   | public   | Track by order ID + phone/email          |
| `POST /api/vendor/apply`                   | auth     | Apply to become a vendor                 |
| `GET/POST/PUT/DELETE /api/vendor/products` | vendor   | Catalogue management                     |
| `GET  /api/vendor/orders`                  | vendor   | Vendor order dashboard                   |
| `PATCH /api/vendor/orders/{id}/fulfillment`| vendor   | Advance fulfillment status               |
| `PATCH /api/vendor/orders/{id}/payment`    | vendor   | Mark payment received                    |
| `GET  /api/vendor/share-link`              | vendor   | Tracked WhatsApp/Instagram share link    |
| `GET  /api/admin/vendors` · approve/reject | admin    | Vendor approval                          |
| `GET  /api/admin/orders`                   | admin    | Platform-wide order oversight            |
| `POST /api/admin/payouts/generate`         | admin    | Vendor payout report                     |
| `GET  /api/meta/option-sets`               | public   | Enum/option sets for the UI              |

## Build & test

```bash
./mvnw test                              # Liquibase + tests on in-memory H2
./mvnw package                           # runnable jar in target/
java -jar target/app-0.0.1-SNAPSHOT.jar
```

## Deployment

`render.yaml` (in this directory) is a Render Blueprint that deploys this API as a
**Docker** service, bound to Render's `$PORT` and health-checked at
`/actuator/health`. Render has no managed MySQL — provision MySQL 8 / TiDB Cloud
separately and set the `DB_*` env vars. Set `APP_CORS_ORIGINS` / `APP_PUBLIC_BASE_URL`
to the frontend's URL.
